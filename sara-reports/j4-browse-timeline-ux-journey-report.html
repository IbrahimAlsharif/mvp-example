USABLE WITH FRICTION

The screen loads, the events show, and a person can read and navigate. But it does not read as the premium, "future-style" life archive the product promises — and that gap is concrete, not taste. The single highest-impact fix:

Highest-impact fix → Replace the empty hour-grid track with a content-dense, year-anchored timeline. Today the signature screen is a large white rectangle holding 2 floating chips against meaningless hour ticks (12:00 · 16:00 · 20:00) — that one element is what makes the whole page feel unfinished and un-premium. Fixing it moves the screen from "friction" toward "strong" more than any other change.

Evidence

Findings

F1 — The signature timeline track is an empty void with the wrong time scale

Method: Visual & IxD (hierarchy, content density) + Nielsen #2 (match the real world) · Nielsen severity 3 · risk

The hero element of a "life timeline" product renders as a large white rectangle (≈846×220px) holding 2 small event chips floating against hour-of-day gridlines (12:00 · 16:00 · 20:00 · 00:00 · 04:00 · 08:00). A life archive spans years; hour ticks are meaningless at this altitude and the emptiness reads as "broken / unfinished," which is exactly the "not future-style" feeling you flagged. This is the dominant reason the page doesn't feel premium.

Fix → Anchor the default view to a year/month density that fills the track: cluster events into period markers, show a faint year/month rail (not hours) in the default zoom, and render an inviting populated state even with few events (e.g. soft "your timeline starts here" markers). Reserve hour ticks for the يوم (day) zoom only.

F2 — Event chip clipped at the track's right edge (RTL overflow)

Method: UX-lens floor (clipping/overflow) + RTL integrity · Nielsen severity 3 · risk

Measured: the تست ٢ 📷 chip ends at x=393 in a 390px viewport (clipped past the right edge) on mobile, and on desktop the second event sits hard against the track's right border. In an RTL layout the most-recent event lands at the right edge and gets cut. Users lose part of an event label — and it looks like a bug.

Fix → Add horizontal padding/inset to the track's content area so chips never touch the edge; clamp chip position to keep the full label inside the visible track, or let the track scroll with the chip fully rendered.

F3 — "+ إضافة حدث" primary action wraps to 3 lines on mobile

Method: UX-lens floor (layout integrity) + Visual hierarchy · Nielsen severity 2–3 · risk

Measured: at 390px the add-event button is 56px tall with line-height 20px — the label breaks across three lines inside an orange pill. The single most important action on the screen looks mangled on phones, which is where parents capture moments.

Fix → Give the button white-space:nowrap and let the header reflow (icon-only on the narrowest breakpoint, or move it to a fixed bottom action bar on mobile).

F4 — Latin date/time labels inside a fully-Arabic RTL UI

Method: Nielsen #4 (consistency) + WCAG 3.1 (language) · Nielsen severity 3 · risk

The whole UI is Arabic (html lang="ar" dir="rtl"), but the timeline axis prints Sun 14 June, Sat 13 June, and 12:00… in English. For the Multi-Generation Family and Elderly personas this is jarring and, for some, unreadable. The native date filters also show dd/mm/yyyy (browser-locale English) instead of an Arabic-formatted hint.

Fix → Format axis dates/times with Intl.DateTimeFormat('ar', …) (Arabic month names, Arabic-Indic or localized digits per product choice). Localize or replace the native date inputs with an Arabic-aware date control.

F5 — Event-count text fails WCAG AA contrast

Method: WCAG 1.4.3 (contrast) · measured ratio 2.52:1 (needs ≥4.5) · Nielsen severity 2 · risk

The "٢ من ٢ حدثاً" count is rgb(163,163,163) at 12px on white — 2.52:1, well below AA. This is meaningful context (how much of your life is captured) rendered nearly invisible, and the Elderly persona will miss it entirely. The like/comment/share labels sit at 4.74:1 — passing but borderline.

Fix → Darken the count to at least #595959 (4.5:1 on white) or bump to 14px+ and a darker weight. Audit all neutral-400 text on white in this view.

F6 — 18 controls below the 44px tap-target minimum on mobile

Method: WCAG 2.5.5 / mobile target size · Nielsen severity 2–3 · risk

At 390px, every filter/view control is 24–32px tall: the circle filters (أنا فقط / العائلة / رابط خاص at 24px), media filters (24px), granularity tabs (28px), and the zoom buttons ＋ − ⛶ (32×32). The journey map names the Elderly Individual as the most at-risk persona; sub-44px targets are exactly where they fail. None are below ~24px (not a hard catastrophe) but the cluster is real friction.

Fix → Raise interactive controls to ≥44×44 hit area on touch (padding is fine; the visual pill can stay smaller). Prioritize the zoom and circle-filter controls.

F7 — Brand title relies on transparent text + background-clip

Method: Visual/IxD robustness · Nielsen severity 1 · advisory

The header brand "الخط الزمني" computes to color: rgba(0,0,0,0) — a gradient clipped to text via -webkit-background-clip. It renders fine in this browser, but if the gradient/clip fails (older engines, forced-colors / high-contrast mode), the title disappears. Low risk; worth a fallback color.

Fix → Set a solid fallback color and guard with @supports for the clipped-gradient treatment.

F8 — Filter controls give no visible "no match" / result feedback

Method: Nielsen #1 (visibility of system status) · Nielsen severity 2, unverified state · advisory (escalate after empty-state pass)

The circle / date / media filters are present, but with only 2 events I could not confirm what happens on an empty result (does the count update? is there a "no events match" message?). Cognitive-walkthrough Q4 ("did the user see progress?") is unanswered for filtering. Flagged for the empty/error-state pass.

Fix → Verify filtered/empty results show an explicit count change and a friendly "no events match these filters" state, not just a blank track.
