# Lesson — Ingest durability is necessary but not sufficient for the decades promise

US-1.2 is the strongest durability spec reviewed so far: it cleanly satisfies H-C1
(confirm gated on checksum-verified persistence, never optimistic) and H-C2 (no
silent drop / no silent partial save). AC-3/4/9 + the telemetry are textbook
"saved means durably persisted + integrity-verified."

**But a well-written ingest-durability story can still pass H-C1/H-C2 and miss the
decades half of the promise.** Two gaps recurred and are candidates for a new
heuristic if they fire again:

1. **Hardware-class durability ≠ memory durability (H-C3).** AC-9 proves byte-intact
   restore after a *single storage-node failure* — the hardware class. It does NOT
   address the ~2/3 of real loss that is human error / app bug / bad release / account
   loss. AWS itself points to versioning + Object Lock + soft-delete for "unintended
   user actions and application failures." A confirmed/persisted blob being recoverable
   after a buggy delete (versioning / soft-delete / recoverable trash) is the larger,
   silent loss vector for a decades product and should be an explicit acceptance
   criterion, not left to G2's prohibition alone.

2. **Ingest fixity ≠ lifetime fixity (H-C4 / NDSA File Fixity / OAIS).** Checksum is
   verified at upload only. The standard expects *periodic re-verification* of stored
   memories to catch bit rot over decades. The minimal non-foreclosing MVP ask: persist
   the client checksum as durable preservation metadata so lifetime fixity (and the
   US-1.4 export fixity manifest) is not foreclosed.

**Proposed heuristic (promote to review-heuristics.md if it fires 3+ times):**
H-C5 — "Durability story must address the human-error/app-bug loss class and persist
fixity metadata, not just survive a storage-node failure at ingest." Fires on any
upload/persistence/durability story. Test: is a persisted item recoverable after a
buggy/accidental delete (versioning/soft-delete), and is the ingest checksum stored as
durable metadata for lifetime re-verification + export? Fail → APPROVED (PROVISIONAL)
with a Confirm-Before-Ship, not a hard block, when ingest durability is otherwise sound.
